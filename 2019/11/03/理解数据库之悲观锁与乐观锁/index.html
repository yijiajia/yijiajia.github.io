Created 2019-11-03| Updated 2019-11-03|数据库
Word count: 884|Reading time: 2 min| Post View:

InnoDB存储引擎

mysql的默认存储引擎，它被设计来处理大量短期事务，支持事务，有自动崩溃恢复特性。InnoDB采用MVCC来支持高并发，并且实现了四个标准的隔离级别。其默认隔离级别为REPEATABLE READ(可重复读)，并通过间隙锁策略防止幻读的出现。在mysql5.6版本后InnoDB是支持全文索引的。

MyISAM存储引擎

在mysql5.1以前是默认的存储引擎，支持全文索引、压缩、空间函数（GIS）等，但不支持事务和行级锁，而且最大的缺陷就是崩溃后无法安全恢复。加锁时会对整张表进行加锁，而不是针对行。在读取时会对所读表进行加共享锁，写入时则对表加排他锁。MyISAM表是可以压缩的，通过myisampack对MyISAM表进行打包压缩，其可以极大地减少了磁盘空间的占用，因此也可以减少磁盘I/O，从而提升查询性能。但是压缩表是不能进行修改的，除非先将表进行解压，修改数据，然后再次压缩。

主要区别

  1. InnoDB支持事务，MyISAM不支持。其中InnoDB的AUTOCOMMIT是默认打开的，即每一条sql语句默认被封装成一个事务，自动提交，这样会影响速度。所以最好将多条sql语句显示放在begin和commit之间，组成一个事务去提交。
  2. MyISAM表可被压缩、存储空间较小。而InnoDB表需要更多的内存和存储，它会在主内存中建立其专用的缓冲池用于高速缓冲数据和索引。
  3. MyISAM只支持表级锁，用户在操作myisam表时，select，update，delete，insert语句都会给表自动加锁，如果加锁以后的表满足insert并发的情况下，可以在表的尾部插入新的数据。而InnoDB支持事务和行级锁，行锁大幅度提高了多用户并发操作的新能。但是InnoDB的行锁，只是在WHERE的主键是有效的，非主键的WHERE都会锁全表的。
  4. InnoDB表获取行数时，需要遍历整个表，消耗非常大，而MyISAM表保存有表的总行数（内置了一个计数器），使用select count(*) from table即可直接取出该值。
  5. MyISAM不支持外键，InnoDB支持。
  6. InnoDB支持热备份

应用场景：

1) MyISAM管理非事务表。它提供高速存储和检索，以及全文搜索能力。如果应用中需要执行大量的SELECT查询，那么MyISAM是更好的选择。
2) InnoDB用于事务处理应用程序，具有众多特性，包括ACID事务支持。如果应用中需要执行大量的INSERT或UPDATE操作，则应该使用InnoDB，这样可以提高多用户并发操作的性能。

但是实际场景中，针对具体问题需要具体分析，一般而言可以遵循以下几个问题：

  • 数据库是否有外键？
  • 是否需要事务支持？
  • 是否需要全文索引？
  • 数据库经常使用什么样的查询模式？在写多读少的应用中还是Innodb插入性能更稳定，在并发情况下也能基本，如果是对读取速度要求比较快的应用还是选MyISAM。
  • 数据库的数据有多大？ 大尺寸倾向于innodb，因为事务日志，故障恢复。
